# /1 — Sprint Start

**Trigger:** Beginning of each sprint. **Prerequisite:** Sprint N code items checked before starting N+1 (developer-parallel tasks excepted).

**Channel:** CT=`CURRENT_TASK_chN.md` (channeled) or `CURRENT_TASK.md`. Scripts: `SENTINEL_CHANNEL=N`. `[chN/]`=dispatch subdir, `[_chN]`=file suffix, `[chN_]`=squad prefix. Full rules: `.claude/reference/channel-routing.md`.

**Step 0:** Before any other work, TaskCreate every step. Mark in_progress→completed.

## Procedure

### Step 1: Spec integrity

User specifies which spec(s). If not specified, ASK.

#### 1a. Dispatch agents

For each spec, count lines and dispatch general-purpose agents (`run_in_background: true`). Explore agents cannot write — do not use them. Substitute ALL placeholders before dispatching: `[SPEC_FILE]`=full path, `[SPEC_NAME]`=filename stem (e.g., `feature_spec`), `[OUTPUT_FILE]`=set per dispatch path below.

**Under 2000 lines:** One agent, full file. `[OUTPUT_FILE]`=`verification_findings/spec_integrity_[SPEC_NAME].md`.

**2000+ lines:** Two-tier dispatch.
a. Grep `## ` headings → section outline with line ranges. (No headings → single agent.)
b. **Tier 1 (structural):** Full file. Prepend: "You are the structural agent reading the full file. Prioritize cross-section contradictions — things chunk agents miss seeing only part of the file." `[OUTPUT_FILE]`=`verification_findings/spec_integrity_[SPEC_NAME].md`.
c. **Tier 2 (detail):** ~1500–2500 line chunks at `## ` boundaries (never mid-section; oversized = own chunk). One agent per chunk. Prepend detail context header. Replace "Read [SPEC_FILE] in full" with "Read [SPEC_FILE] from line [START] to line [END]". Additional: `[START]`/`[END]`=chunk range, `[TOTAL]`=total lines, `[N]`=1-based index, `[SECTION_NAMES]`=assigned headings. `[OUTPUT_FILE]`=`verification_findings/spec_integrity_[SPEC_NAME]_part[N].md`.

#### 1b. Parent direct verification (while agents run)

- **Tables:** Verify all arithmetic (row/column sums, totals).
- **Cross-references:** Grep spec references to other files → verify each exists on disk.
- **Enums/constants:** Grep spec enum values → compare against code definitions. Flag mismatches.

#### 1c. Consolidate

Read all agent outputs. Two-tier: cross-check "PARTIAL CONTEXT" items against other agents or by reading directly. Deduplicate with 1b findings. Remove confirmed false positives. Write to `verification_findings/spec_integrity_[SPEC_NAME].md`.

#### 1d. Present findings

Show consolidated findings. Do NOT auto-fix or suggest fixes unless asked. User decides what to fix. **User gate.**

### Step 2: Staleness scan + dependency verification

**MANDATORY SONNET DELEGATION.** All three agents are mechanical grep/filesystem verification.

Write single Sonnet prompt to `verification_findings/_pending/[chN/]sprint_start_scan_<timestamp>.md`. Wait: `bash scripts/wait_for_results.sh verification_findings/sprint_start_infrastructure[_chN].md verification_findings/sprint_start_specs[_chN].md verification_findings/sprint_start_deps[_chN].md`.

- **Agent A — Infrastructure:** Read every sprint-dependent file (engines, providers, schema, services). Report per file: exists/stubbed/missing, line counts, TODO markers. → `verification_findings/sprint_start_infrastructure[_chN].md`.
- **Agent B — Specs:** Read every referenced spec. Verify content counts, terminology (`.claude/reference/terminology.md`), cross-references, no stale headers. → `verification_findings/sprint_start_specs[_chN].md`.
- **Agent C — Dependencies:** Prerequisite sprints merged with passing tests. Developer tasks complete or explicitly non-blocking. → `verification_findings/sprint_start_deps[_chN].md`.

Read results. Fix stale items. Note already-accomplished. Fix terminology violations on sight.

### Step 3: Write INITIAL CT

Include: infrastructure table (what's built, paths, DO NOT REBUILD), spec status (specified vs. gaps), known tasks, key files table (paths + why).

### Step 4: Phase-gate verification

**MANDATORY SONNET DELEGATION.** Fact-checking claims is mechanical.

Write Sonnet prompt to `verification_findings/_pending/[chN/]sprint_gate_<timestamp>.md`. Agent checks every CT claim against disk. Output: `verification_findings/sprint_start_gate[_chN].md`. Wait: `bash scripts/wait_for_results.sh verification_findings/sprint_start_gate[_chN].md`. Fix discrepancies immediately.

### Step 5: Present for user approval

Show CT. Announce `/1` complete. Do NOT proceed to `/2` until user approves.

## Agent prompt template

```
Read project-specific rules files (design invariants, terminology, etc.).

Read [SPEC_FILE] in full. For each section in your assigned range, run
checks 1–6 below. Write results to [OUTPUT_FILE] using the format at the end.

--- CHECK 1: COUNT AND CATALOG ---

Count discrete items per section (features, parameters, UI elements, stages,
steps). Record exact counts. These anchor every other check.

--- CHECK 2: INTERNAL CONSISTENCY ---

The spec must agree with itself. Flag:
- A value, term, or description that appears in multiple places with different
  content (a theme in a definition vs the same theme in a summary table; a
  parameter in a formula vs its value in a config section)
- A design rule stated in one section and violated in another
- Stale references to items that were moved, renamed, or deleted
- Content from one module's domain appearing in another module's flow when the
  spec's own design principles prohibit it

Quote both sides of every contradiction with line numbers.

--- CHECK 3: SOURCE VERIFICATION ---

Every technique, citation, and attributed claim must be traceable to a primary
source. Flag:
- Citations you cannot verify (author + year + title)
- A real citation used to support a claim the cited work did not make — this
  is a fabricated application and is MAJOR severity, not minor
- Technique names that don't match their stated tradition or lineage
- Claims that sound precise and authoritative but may be invented

When in doubt, flag. False positives cost less than missed hallucinations.

--- CHECK 4: DESIGN INVARIANTS ---

You read project rules files above. Check every section against ALL
invariants in those files, then run mechanical grep verifications on
[SPEC_FILE] (within your assigned line range only) for any project-specific
patterns defined in the rules.

Do not rely on reading alone for word-level violations. Grep catches what
reading misses.

--- CHECK 5: CROSS-FILE REFERENCES ---

Verify every referenced file path and section name exists using Grep/Glob.
Flag fabricated references — plausible-sounding paths or section names that
point to nothing.

--- CHECK 6: CODE CROSS-REFERENCES ---

For specs with structured IDs or enums:
- Verify EACH ID exists in the corresponding code file (grep exact string)
- For EACH item with metadata: compare spec value against code. Flag any
  mismatch.

For specs with structured parameters (timing, thresholds, stage criteria):
- Verify parameter values match any corresponding code constants

--- OUTPUT FORMAT ---

Write to [OUTPUT_FILE]:

  ## [Section Name]
  Items counted: N
  Contradictions: [list or "none"]
  Source issues: [list or "none"]
  Invariant violations: [list or "none"]
  Cross-file issues: [list or "none"]
  Code mismatches: [list or "none"]

You MUST write to the file before returning. Do not return findings only
in context.
```

## Detail context header

Prepend to the agent prompt template for Tier 2 detail agents.

```
SCOPE: You are auditing lines [START]–[END] of [SPEC_FILE] ([TOTAL] lines total).
Other agents cover the rest in parallel.

SECTION OUTLINE (full file — for reference, not for processing):
[PARENT_INSERTS_SECTION_HEADINGS_WITH_LINE_RANGES]

YOUR SECTIONS: [SECTION_NAMES]
Process ONLY the sections listed above. "For each section" in the checklist
means each section in your assigned range, not every section in the outline.

PARTIAL CONTEXT RULE: If you encounter a reference to content outside your
line range, mark it "PARTIAL CONTEXT — outside my range, parent will cross-check"
instead of flagging it as missing or broken.
```
